# 先验证路线：整体架构方案

## 融合 seL4 精化链 × HoarePrompt 结构化推理 × SysSpec 规格驱动生成

---

## 1. 核心理念

**先写规格、先完成验证，再由 LLM 生成满足规格的代码。**

整条链路的验证骨架复用 seL4 的精化链条（ASpec → ExecSpec → C Spec），精化关系在定理证明器（Lean / Coq / Isabelle，工具选择开放）中做机器检查。LLM 在链路的每一层承担生成任务，定理证明器承担检查任务。

---

## 2. 端到端链路总览

```
设计需求（自然语言 / 半形式化）
        │
        ├─── 分支一：需要建模的需求（系统语义性 / 模糊性）
        │         │
        │         ▼
        │    LLM 建模 + HoarePrompt 式半形式化推理
        │         │
        │         ▼
        │    抽象规范（= 形式化模型）
        │
        ├─── 分支二：完整数学对象
        │         │
        │         ▼
        │    LLM 翻译 → 形式化需求（纯数学定义，定理证明器中）
        │         │                        ▲
        │         │                        │ 锚点：终止状态必须满足此定义
        │         ▼                        │
        │    LLM 生成 → 抽象规范（算法级状态机）
        │
        └────────┬───────┘
                 │  ← 抽象规范自我一致性验证（融合在此迭代中）
                 ▼
           抽象规范（已验证一致性）
                 │
                 ▼
           三者 Loop（复用 seL4 架构）：
           LLM 生成函数实现原型 ←→ 动态验证（执行测试）
                 │
                 ▼ 自动翻译
           执行层规约（定理证明器中）
                 │
                 ▼ 精化证明：执行层规约 精化 抽象规范
           ─────────────────────────
                 │
                 ▼
           三者 Loop（同构结构）：
           LLM 生成 C/Rust 代码 ←→ 动态验证（编译 + 测试）
                 │
                 ▼ 自动翻译（c-parser / 语义工具）
           C/Rust Spec（定理证明器中）
                 │
                 ▼ 精化证明：C/Rust Spec 精化 执行层规约
           ─────────────────────────
                 │
                 ▼
           已验证的 C/Rust 实现
```

## 3. 设计需求 → 抽象规范

### 3.1 两个分支

**分支一：需要建模的需求**

适用于系统语义性需求（"中断到来时通知 handler"、"capability 不被未授权访问"）和模糊性需求（需要先建模才能形式化的场景）。

- 抽象规范本身就是建模的产物
- "模型是否忠实反映需求"不可完全形式化验证
- 信任边界在此：抽象规范是验证链的顶端锚点，其正确性依赖人类审查

**分支二：完整数学对象**

适用于背后有严格数学定义的需求（因式分解、排序、密码学原语等）。

- 多一层工件：**形式化需求**（纯数学定义，在抽象规范之上）
- LLM 做的是"把已有数学定义翻译成定理证明器语言"
- 翻译结果可由定理证明器做 type check 和关键性质证明
- 信任边界消失：从顶到底全链路机器检查

### 3.2 统一的迭代环

两个分支共享同一个迭代结构，包含三个反馈通道：

```
LLM 生成抽象规范候选（状态空间 + 不变量 + 操作语义）
        │
        ├─→ 定理证明器检查（硬信号）
        │     • 初始状态存在性（不变量非空）
        │     • 各操作保持不变量
        │     • 前条件可达性
        │     • [分支二] 终止状态满足形式化需求
        │     → 失败则反馈具体证明错误
        │
        ├─→ LLM 半形式化推理（软信号，HoarePrompt 式）
        │     • 在关键场景下展开状态转换
        │     • 检查行为是否符合设计需求意图
        │     → 发现不合理则标记问题
        │
        ├─→ 人类审查（兜底）
        │     • 建模是否忠实反映需求（分支一核心）
        │     • 形式化需求翻译是否正确（分支二辅助）
        │
        └─→ LLM 根据反馈修改 → 下一轮迭代
```

三类信号驱动同一个修改循环，不是三个独立的环。

### 3.3 验证强度差异

| | 分支一 | 分支二 |
|---|---|---|
| 顶端锚点 | 抽象规范本身（人类审查） | 形式化需求（数学定义） |
| 需求→规范 | 半形式化论证 + 人审 | 机器检查 |
| 规范往下 | 全链路机器检查 | 全链路机器检查 |
| 信任假设 | 规范忠实反映意图 + 证明器正确 | 证明器正确 |

## 4. 抽象规范的内容结构

抽象规范统一以**状态机**为骨架组织（包括分支二：数学对象的实现也需要计算流程，流程就是状态机）。

### 4.1 组成部分

| 组成部分 | 说明 | seL4 对应 |
|---|---|---|
| **状态空间定义** | 抽象数据模型，用集合、映射、树、代数结构等数学对象表达，不涉及具体数据结构 | `Structures_A`, `ARM_Structs_A` |
| **状态的数学性质** | 静态约束（单状态必须满足）+ 动态不变量（所有操作必须保持）。包括代数性质、序关系、守恒量等 | `proof/invariant-abstract/` |
| **操作/转换语义** | 状态机的转换关系，每个操作定义为状态→新状态（可含非确定性） | `Syscall_A` + 各子系统理论 |
| **操作的正确性性质** | 每个操作的前后条件，及操作对状态数学性质的保持性（Hoare 三元组） | `invariant-abstract/` 中的证明 |
| **全局正确性性质** | 安全性质、活性性质、终止条件；分支二额外包含终止状态对形式化需求的符合性 | `access-control/`, `infoflow/` |
| **模块边界与接口** | 多模块系统的模块间接口关系（可选，借鉴 SysSpec 的 rely-guarantee） | 无直接对应 |
| **非确定性与抽象选择** | 显式标明哪些行为留给下层决定 | nondet_monad 集合语义 |

### 4.2 与 seL4 的结构差异

seL4 把"状态的数学性质"和"操作保持不变量的证明"放在 `proof/` 目录而非 `spec/` 目录——ASpec 只定义语义，性质证明是额外的。我们的框架把性质陈述纳入抽象规范，使规范"更厚"：既包含语义定义，也包含性质陈述和证明义务。

## 5. 抽象规范 → 执行层规约

### 5.1 这一步做什么

将抽象规范中的数学对象具体化为可实现的算法和数据结构，但还在定理证明器中，不是目标语言代码。

核心设计决策：
- **数据结构选择**：集合→哈希表/平衡树，树→链表/数组索引，等
- **算法选择**：从"找到满足条件的元素"到具体搜索/遍历算法
- **非确定性消解**：抽象层保留的非确定性选择在此固定
- **模块分解细化**：原子转换拆成子步骤和模块间调用序列

### 5.2 三者 Loop（复用 seL4 架构）

```
              抽象规范
                 │
       ┌─────────┼─────────┐
       ▼         │         │
  函数实现原型    │    精化证明
  （可执行语言） │   （定理证明器）
       │         │         ▲
       ▼         │         │
  动态验证  ─────┘    执行层规约
  （跑测试）      自动翻译 ▲
       │         ─────────┘
       ▼
  快速反馈 → LLM 修改原型 → 下一轮
```

三个工件的角色：

| 工件 | 角色 | seL4 对应 |
|---|---|---|
| 函数实现原型 | 可执行设计模型，用于快速迭代和动态验证 | Haskell 原型 |
| 执行层规约 | 由原型自动翻译到定理证明器，不手改 | ExecSpec（自动生成） |
| 精化证明 | 证明执行层规约精化抽象规范 | `proof/refine/` |

### 5.3 迭代过程

1. LLM 从抽象规范生成函数实现原型（借鉴 SysSpec SpecCompiler 策略）
2. 动态验证：跑原型，用测试用例快速筛掉明显错误
3. 自动翻译：原型 → 执行层规约（翻译器保证语义一致）
4. 形式化验证：定理证明器尝试证明精化关系
5. 失败 → 反馈（状态关系 R 不当 / 操作语义不一致）→ LLM 修改原型
6. 回到步骤 2

动态验证先收敛，再启动形式化验证——避免浪费证明器时间在明显不对的实现上。

### 5.4 借鉴 SysSpec 的生成策略

- **两阶段生成**：先逻辑（数据结构 + 算法），再处理其他维度（如并发，若有）
- **模块独立生成**：按 rely-guarantee 接口分模块，LLM 每次只生成一个模块
- **retry-with-feedback**：验证失败时带具体错误信息重试

### 5.5 关键纪律

**LLM 只改原型，不直接改执行层规约。** 执行层规约由翻译器生成，保证"动态验证通过的原型"和"精化证明针对的执行层规约"是同一个东西。

## 6. 执行层规约 → C/Rust 实现

### 6.1 同构的三者 Loop

```
            执行层规约
                 │
       ┌─────────┼─────────┐
       ▼         │         │
  C/Rust 代码    │    精化证明
       │         │   （定理证明器）
       ▼         │         ▲
  动态验证  ─────┘    C/Rust Spec
  （编译+测试）   自动翻译 ▲
       │         ─────────┘
       ▼
  快速反馈 → LLM 修改代码 → 下一轮
```

| 工件 | 角色 | seL4 对应 |
|---|---|---|
| C/Rust 代码 | 目标实现，可编译运行 | C 实现（seL4/ 仓库） |
| C/Rust Spec | 自动翻译到定理证明器的语义表示 | C Spec（c-parser 生成） |
| 精化证明 | 证明 C/Rust Spec 精化执行层规约 | `proof/crefine/` |

### 6.2 这一层必须面对的实现细节

执行层规约还在定理证明器的"纯净世界"里——没有指针、没有内存布局、没有并发原语、没有硬件交互。到 C/Rust 则进入"真实硬件世界"，以下维度必须在这一层做出明确决策：

**内存模型**

执行层规约操作的是抽象数据结构（列表、树、映射），到 C/Rust 要落到具体内存布局。需要明确：堆分配 vs 栈分配策略、对象生命周期管理、指针关系与所有权、内存对齐要求、数据结构的内存表示（packed struct、tagged union 等）。seL4 为此专门写了 bitfield generator 来处理内核数据结构到硬件内存布局的映射。Rust 的所有权系统能在类型层面约束一部分（生命周期、借用规则），C 则完全靠显式管理。

**并发实现**

如果系统有并发语义，上层可能用 rely-guarantee 抽象描述了模块间交互约束，到这一层要落到具体同步原语：互斥锁、读写锁、原子操作、无锁数据结构、中断使能/禁用策略。需要明确锁粒度（粗粒度简单但限制并发度，细粒度高效但证明复杂）、死锁避免策略（锁序、trylock）、中断上下文与线程上下文的交互规则。seL4 的策略是把并发压到最小（单处理器 + 关中断 + 事件驱动），但不是所有系统都能这样做。

**性能优化**

缓存友好的数据布局（结构体字段排列、热/冷数据分离）、热路径内联、循环展开、分支预测友好的代码组织（likely/unlikely 提示）、避免不必要的内存拷贝（零拷贝设计）、SIMD 向量化等。每个优化都是对执行层规约的"语义保持但结构偏离"，都需要在精化证明中额外处理。优化越激进，精化证明越重。

**硬件交互**

设备寄存器访问（MMIO）、DMA 缓冲区管理、中断处理路径、平台特定指令（内存屏障、缓存控制、TLB 管理等）。seL4 把这些压到极少量汇编里并明确验证边界。对其他系统，可能需要更多的硬件抽象层设计。

**错误处理**

执行层规约可能用代数类型（Result/Option/Either）表达错误路径，到 C 要变成错误码返回、errno 设置、goto 清理路径等命令式错误处理模式。Rust 的 Result 类型与执行层规约更接近，这是 Rust 在这一层的一个优势。

**语言范式转换**

代数数据类型 → struct/enum、模式匹配 → if-else/switch、递归 → 循环（可能需要显式栈）、高阶函数 → 函数指针或内联展开、不可变数据 → 可变状态与原地更新。

### 6.3 生成策略

这些维度不应该一次性全部处理。借鉴 SysSpec 的经验，采用**多阶段生成**：

1. **第一阶段：逻辑正确的直译** —— 先生成功能正确但未优化的 C/Rust 代码，尽可能保持与执行层规约的结构对应。此阶段精化证明最容易。
2. **第二阶段：内存模型落地** —— 明确内存布局、分配策略、生命周期管理。
3. **第三阶段：并发实现** —— 引入同步原语，落实锁协议。
4. **第四阶段：性能优化** —— 在已验证正确的基础上逐步引入优化，每次优化后重新验证精化关系。

先生成逻辑正确的代码再逐步优化，比一步到位生成优化代码的成功率高得多。每个阶段都走一遍"生成 → 验证 → 反馈 → 修改"的 loop，确保每一步变换都不破坏精化关系。

## 7. LLM 与定理证明器的职责边界

| | LLM 负责 | 定理证明器负责 |
|---|---|---|
| 设计需求→抽象规范 | 建模、翻译、HoarePrompt 式推理 | 一致性检查（不变量可满足、操作保持不变量等） |
| 抽象规范→执行层规约 | 生成函数实现原型 | 精化证明（ExecSpec 精化 ASpec） |
| 执行层规约→C/Rust | 生成目标代码 | 精化证明（C/Rust Spec 精化 ExecSpec） |
| 精化证明本身 | 生成证明骨架 / tactic 脚本候选 | 机器检查所有证明 |

**原则：LLM 生成，证明器检查。LLM 的输出不被信任，一切以证明器的判定为准。**

## 8. 整体验证强度

完整精化链：

```
[分支二] 形式化需求
            │ 定理证明器检查
            ▼
       抽象规范  ←── [分支一] 人类审查（信任假设）
            │ 精化证明（机器检查）
            ▼
       执行层规约
            │ 精化证明（机器检查）
            ▼
       C/Rust Spec
            │ = 真实代码的语义
            ▼
       C/Rust 实现
```

在抽象规范上证明的安全性质、正确性性质，通过精化链传递到真实实现。

信任假设（TCB）：
- 定理证明器正确
- 自动翻译器（原型→执行层规约、C/Rust→Spec）语义保持
- 编译器正确（或使用验证编译器如 CompCert）
- 硬件正确
- [仅分支一] 抽象规范忠实反映设计意图

## 9. 三个参照系的融合总结

| 参照系 | 贡献 | 在架构中的位置 |
|---|---|---|
| **seL4** | 精化链条方法论（多层规范 + 逐层精化 + 机器检查）；三者 loop（原型 ↔ 规约 ↔ 证明） | 整体骨架 |
| **HoarePrompt** | 结构化自然语言推理（最强后条件式状态传播） | 设计需求→抽象规范的半形式化推理环节 |
| **SysSpec** | 多部分规格框架（功能/模块性/并发）；SpecCompiler 两阶段生成；retry-with-feedback 闭环 | 规格组织模板 + 代码生成策略 |
